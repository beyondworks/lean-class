# Model A/B Test — Opus 4.8 vs Sonnet 5

## 분석의 초점 (가장 중요)
이 스킬은 **결과물 품질 우열을 가리는 게 아니다.** 초점은 **각 모델이 "어떻게 사고하고 일하는가"** —
모델 자체의 행동·인지 특성이다. 항상 아래를 본다:

- **문제 분해** — 잘게 쪼개는가, 통째로 가는가. 어떤 단위로 나누는가.
- **추론 경로** — 선형적으로 밀고 가는가, 여러 갈래를 탐색·백트래킹하는가.
- **일 푸는 순서·과정** — 먼저 계획을 세우는가 즉흥 착수하는가. 어떤 스텝을 거치는가.
- **가정 처리** — 가정을 명시하는가 암묵으로 두는가. 불확실성을 표기하는가.
- **도구 사용 전략** — 언제, 왜 도구를 쓰는가. 결과를 검증하는가.
- **자기교정** — 막히거나 틀렸을 때 스스로 잡는가. 어떻게 되돌아오는가.
- **탐색 vs 활용** — 여러 대안을 열어보는가, 첫 해법을 밀어붙이는가.
- **지시 해석** — 문자 그대로 따르는가, 의도를 추론해 채우는가.
- **토큰 배분·장황함** — 어디에 말을 쓰는가(계획/설명/검증 중).

결과물은 이 특성을 읽어내기 위한 **증거**로만 쓴다.

---

## 두 가지 모드

- **실행 모드** — `/model-ab-test <프롬프트>` : 한 프롬프트를 두 모델로 실행하고 **과정 특성**을 비교·로그.
- **리포트 모드** — `/model-ab-test insight|done|report` (또는 "인사이트", "분석", "리포트") : 누적 로그를 **최상위 모델로 종합 분석**해 리포트를 낸다.

---

## A. 실행 모드

### 1. 프롬프트 확정 (변수 통제)
비교 대상 프롬프트를 고정. **두 실행에 완전히 동일한 프롬프트**를 준다(문구를 모델별로 바꾸지 않는다).

### 2. 두 모델로 병렬 실행 — 기본: 서브에이전트 분기
**한 메시지 안에서 Agent(Task) 도구를 두 번 동시 호출**. 프롬프트 동일, `model`만 다르게:

- 에이전트 A: `model: "opus"` (→ Opus 4.8)
- 에이전트 B: `model: "sonnet"` (→ Sonnet 5)
- 둘 다 같은 지시 + **과정 자기보고 요구**를 덧붙인다:
  > "<프롬프트>
  >
  > 작업을 수행하라. 그리고 결과물과 **별도로**, 네가 이 문제를 어떻게 풀었는지 과정을 밝혀라:
  > (1) 문제를 어떻게 분해했나 (2) 어떤 순서로 접근했나(먼저 계획했나/즉흥이었나)
  > (3) 무슨 가정을 두었나 (4) 도구를 언제·왜 썼나 (5) 막힌 지점과 되돌아온 방식
  > (6) 대안을 몇 개나 열어봤나. 과정 보고는 솔직하게, 사후 미화 없이."
- `subagent_type`은 태스크에 맞게(`general-purpose`/`Explore`/`code-reviewer` 등).
- **과정 신호도 함께 수집**: 각 에이전트 완료 시 리턴되는 `tool_uses`, `duration_ms`, 반환 길이 —
  이것도 "일하는 방식"의 정량 단서다(도구를 많이/적게 쓰는가, 빠른가).

**폴백 — 완전 격리 필요 또는 오버라이드 미작동 시**: Bash 백그라운드 헤드리스,
`--verbose`로 사고·도구 흐름을 캡처:
```
claude -p "<프롬프트>" --model claude-opus-4-8  --verbose > opusA.txt 2>&1   # run_in_background
claude -p "<프롬프트>" --model claude-sonnet-5  --verbose > sonnetB.txt 2>&1  # run_in_background
```

### 3. 과정 비교
두 결과·과정보고·정량신호를 위 "분석의 초점" 9개 축으로 대조. **우열이 아니라 스타일의 차이**를 짚는다.
근거를 대고, 추정은 "추정"으로 표기. (예: "Opus는 착수 전 3단계 계획을 세운 뒤 진행한 반면,
Sonnet은 바로 코드부터 쓰고 뒤에 검증했다" 같은 과정 서술.)

### 4. 로그 기록 (필수)
`~/.claude/logs/model-ab-test.md` 끝에 **append**(추가만). 결과 요지보다 **과정 특성**을 남긴다:
```
### YYYY-MM-DD HH:MM | A/B | <프롬프트 요약>
- 프롬프트: (원문/요약)
- Opus 4.8 — 일하는 방식: (분해·순서·계획성·도구전략·자기교정·탐색성 관찰) / 정량: tool_uses·duration
- Sonnet 5 — 일하는 방식: (동일 축) / 정량: tool_uses·duration
- 과정 차이: (두 모델의 접근이 어디서 갈렸나 — 근거와 함께, 추정은 표기)
- 세션: (프로젝트 경로)
```
**시크릿 평문 금지.**

### 5. 보고
두 결과 요약 + **과정 차이 3줄** + 로그 기록 완료 알림.

## B. 리포트 모드 (`insight` / `done` / `report`)

누적 로그를 종합해 **"두 모델이 각각 어떻게 사고하고 일하는가"**의 큰 그림을 리포트로 낸다.

### 1. 로그 로드
`~/.claude/logs/model-ab-test.md` 전체를 읽는다. 엔트리 수·태스크 다양성을 먼저 센다(표본 파악).

### 2. 분석 위임 — 최상위 모델 + 리서치 사이언티스트 페르소나
Agent 도구를 `model: "opus"`(현재 클로드 최상위 = Opus 4.8)로 분기해 분석을 맡긴다.
현재 세션이 이미 Opus 4.8이어도, 분석은 **전용 분석 에이전트**로 분리해 집중시킨다.
분석 에이전트에 넘길 지시:

> "당신은 실리콘밸리 프런티어 AI 랩의 **시니어 리서치 사이언티스트**다. SOTA 모델의 행동을
> 평가하고 학술 논문을 집필하는 수준의 분석력을 가졌다. 아래 A/B 테스트 로그를 분석하라.
>
> **분석 초점은 결과 품질이 아니라 '각 모델이 어떻게 사고하고 일하는가'다** — 문제 분해 방식,
> 추론 경로(선형/탐색·백트래킹), 계획성, 가정·불확실성 처리, 도구 사용 전략, 자기교정,
> 탐색 대 활용, 지시 해석, 토큰 배분. 두 모델의 **인지적·행동적 지문(behavioral fingerprint)**을
> 뽑아내라.
>
> 도출할 것: (1) 각 모델의 일하는 방식·추론 스타일을 한 문단으로 특징화 (2) 태스크 유형별로
> 갈리는 지점 (3) 반복되는 행동 패턴과 그 이유 가설 (4) '이런 일에는 이 모델' 실무 결정 규칙
> (5) 표본 한계·신뢰도와, 특성을 더 선명히 하려면 다음에 무엇을 테스트해야 하는지.
>
> 규율: 로그 근거로만 말하라. 근거가 얇으면 '표본 부족'을 명시하고 과일반화하지 마라.
> 관찰과 추정을 구분해 표기하라.
>
> **문체(엄수): 분석은 깊게 하되, 리포트 문장은 비전문가도 읽는 쉬운 한국어로 써라.**
> 전문용어·수식·영어 약어를 최소화하고, 꼭 필요하면 한 줄로 풀어 설명하라. 어려운 말로
> 똑똑해 보이려 하지 말고, 쉬운 말로 정확하게 짚어라."

### 3. 리포트 구성 (분석 에이전트가 채움)
1. **한눈 요약** — 두 모델의 일하는 방식을 각각 2~3문장으로.
2. **모델별 행동 지문** — Opus 4.8 / Sonnet 5 각각: 분해·추론경로·계획성·도구전략·자기교정·탐색성.
3. **어디서 갈리나** — 태스크 유형별(코딩·글쓰기·분석·탐색·긴 맥락 등) 접근 차이.
4. **실무 결정 규칙** — "이런 상황엔 이 모델" (근거와 함께).
5. **표본과 한계** — 엔트리 수, 편향, 다음에 테스트할 것.

### 4. 저장 + 보고
리포트를 `~/.claude/logs/model-ab-test-reports/YYYYMMDD-HHMM.md`에 저장(디렉토리 없으면 생성).
사용자에게 **한눈 요약 + 실무 결정 규칙**을 제시하고 리포트 파일 링크를 준다.

## 주의
- **부작용 있는 태스크(파일 쓰기·외부 발송·삭제·배포)는 실행 모드로 돌리지 않는다.** 읽기·생성·분석·설계만.
  불가피하면 드라이런(계획만)으로 비교.
- 모델 이름 매핑: `opus`=`claude-opus-4-8`, `sonnet`=`claude-sonnet-5`. 리포트 분석은 항상 최상위(`opus`).
- 검증됨(2026-07-01): Agent `model:"opus"`/`"sonnet"` 오버라이드가 실제로 각각
  `claude-opus-4-8`/`claude-sonnet-5`를 띄우는 것 실측 확인 → 서브에이전트 분기 경로 신뢰 가능.
- 두 서브에이전트가 파일을 동시에 쓰면 충돌 — 쓰기 태스크면 `isolation: "worktree"` 또는 폴백 경로.
